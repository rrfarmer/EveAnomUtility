# EveJS Content Customization Research

Date: 2026-06-18

## Server-Side Authority Layers

EveJS already has the right server-side layers for a content admin utility:

1. Static/reference tables that should generally be read-only in this tool:
   - `solarSystems`, `stargates`, `itemTypes`, `shipTypes`, `typeDogma`, `shipDogmaAttributes`, `stations`, `celestials`.
   - These provide names, positions, security, client-compatible type IDs, and dogma-derived behavior.

2. Server-authored/synthetic content authority:
   - `dungeonAuthority`: site templates, site families, scene profiles, population hints, gates, resources, encounters, containers, hazards, and universe spawn profiles.
   - `missionAuthority`: client mission metadata, mission IDs, content templates, localized names/messages, mission indexes.
   - `agentAuthority`: agent records and mission pools that link agents to mission content.
   - `npcProfiles`, `npcLoadouts`, `npcBehaviorProfiles`, `npcLootTables`, `npcSpawnPools`, `npcSpawnGroups`, `npcSpawnSites`, `npcStartupRules`.

3. Runtime/player-owned state:
   - `dungeonRuntimeState`, `missionRuntimeState`, `miningRuntimeState`, `probeRuntimeState`, `wormholeRuntimeState`, `npcRuntimeState`, `npcControlState`, `items`, `characters`, and related player/runtime tables.
   - These must only be modified in a cloned data root while developing and testing.

4. Generated code-backed rows:
   - CONCORD default startup rules are generated from config and authored-rule coverage in `server/src/space/npc/npcDefaultConcordRules.js`.
   - EverMore Jita gate profiles/loadouts/spawn group are generated by `server/src/space/npc/empireSecurity/empireSecurityNpcCatalog.js` and layout constants in `server/src/space/empireGatePresence/everMoreGatePresence.js`.
   - Capital and Trig/Drifter NPC catalogs also contribute generated rows to the NPC lookup layer.

## Server Hooks And Paths

### Database Safety

- `server/src/newDatabase/index.js` supports `EVEJS_NEWDB_DATA_DIR`.
- EveJS resolves its runtime data root as: `EVEJS_NEWDB_DATA_DIR` if set, otherwise `_local\newDatabase\data` when `_local\newDatabase\manifest.json` or the local data folder exists, otherwise `server\src\newDatabase\data`.
- In the current checkout, `_local\newDatabase\data` is the real runtime database. `server\src\newDatabase\data` contains many table controller shims and only some source authority JSON, so cloning that source folder produces incomplete catalog data.
- The admin utility should clone the resolved runtime data root into `D:\EveAnomUtility\workspace\db-clone`. Current resolver target: `C:\Users\ryanf\Documents\GitHub\eve.js\_local\newDatabase\data`.
- Never write to the live EveJS runtime data directory during development.
- Generated template packs should use current `dungeonAuthority.templatesByID` row shape: top-level `source`, `sourcePriority`, `sourceConfidence`, `siteFamily`, `siteKind`, `siteOrigin`, `resolvedName`, `resourceComposition`, structured `populationHints`, and optional utility-specific detail under `adminMetadata`.

### Dungeon/Anomaly Runtime

- `server/src/services/dungeon/dungeonAuthority.js` indexes `dungeonAuthority` by `templateID`, `siteFamily`, source dungeon ID, archetype, resource type, and spawn profile family.
- `server/src/services/dungeon/dungeonRuntime.js` creates and reconciles runtime dungeon instances.
- `server/src/services/dungeon/dungeonUniverseRuntime.js` builds desired universe site definitions and generated mining definitions.
- `server/src/services/dungeon/dungeonUniverseSiteService.js` materializes site roots, containers, hazards, acceleration gates, environment props, objectives, NPC encounter plans, loot, and completion.
- `server/src/services/dungeon/dungeonInstanceCacheMgrService.js` feeds Agency/map-style caches for combat, ore, ice, faction warfare, Homefront, pirate insurgency, Triglavian, signatures, and Bulwark mining counts.
- `server/src/services/exploration/signatures/signatureRuntime.js` and provider modules feed live scanner state.

### Mission Runtime

- `server/src/services/agent/missionAuthority.js` indexes mission authority and template-to-mission mappings.
- `server/src/services/agent/agentMissionRuntime.js` chooses mission templates, creates private mission dungeon instances, creates mission bookmarks, builds journal/objective payloads, materializes mission sites, and completes/cleans up accepted missions.
- Mission combat should be edited through mission/dungeon presentation templates and mission runtime dungeon links, not as public scanner anomalies.
- Mission subtype distinction in cloned `missionAuthority`:
  - Combat missions have `missionKind: "encounter"` and/or a `killMission` object. `killMission.dungeonID` maps to the dungeon template ID `client-dungeon:<dungeonID>`.
  - Courier missions have `missionKind: "courier"` and a `courierMission` objective item/quantity. They do not expose a combat dungeon template.
  - Trade missions often also have a `courierMission` object because the server needs an item-delivery objective, but they must remain `missionKind: "trade"` rather than being folded into courier.
  - Mining missions can carry site/objective helper data, but explicit `missionKind: "mining"` wins over helper objects.
  - Current cloned counts: 2,879 missions total; 1,617 combat, 617 courier, 42 mining, 534 trade, 49 talk-to-agent, 20 agent-interaction.

### NPC Runtime

- `server/src/space/npc/npcData.js` composes authored rows plus generated rows. The safe customization ladder is:
  - `npcProfiles`: name, presentation type, faction/corp, loadout ID, behavior profile ID, loot table ID, bounty, spawn defaults.
  - `npcLoadouts`: module type IDs, quantities, charges, cargo.
  - `npcBehaviorProfiles`: think interval, auto-aggro, target preference, orbit/follow/leash ranges, e-war/repair/drifter/capital behavior flags.
  - `npcLootTables`: loot behavior.
  - `npcSpawnPools`: weighted profile selection.
  - `npcSpawnGroups`: fixed or pooled group entries.
  - `npcSpawnSites`: named spawn site wrappers, currently empty in local data.
  - `npcStartupRules`: system/gate/station startup placement rules.
- `server/src/space/npc/nativeNpcService.js` materializes native NPC entity/controller/module/cargo records.
- `server/src/space/npc/npcEquipment.js` derives weapon/e-war/repair/superweapon behavior from fitted modules and Dogma where possible.

### CONCORD And EverMore

- Passive generated CONCORD gate/station presence:
  - Config keys in `server/src/config/index.js` include `npcDefaultConcordStartupEnabled`, `npcDefaultConcordGateAutoAggroNpcsEnabled`, `npcDefaultConcordStationScreensEnabled`, `npcDefaultEverMoreGatePresenceEnabled`, and `npcDefaultEverMoreGatePresenceSystemIDs`.
  - `npcDefaultConcordRules.js` generates high-sec gate/station startup rules unless authored rules already cover a system.
  - EverMore replaces generic Jita gate CONCORD when enabled and the system passes `isEverMoreGatePresenceSystem`.

- Crimewatch CONCORD response:
  - `server/src/services/security/crimewatchState.js` schedules punitive high-sec response based on criminal actions.
  - It chooses `concord_crimewatch_response_5/6/7` spawn groups by system security and spawns transient responders through `npcRuntime.spawnGroupInSystem`.
  - This should be an advanced law-enforcement behavior panel, separate from gate/station ambiance.

## Safe Customization Boundary

Allowed server-side customization:

- Create overlay site templates for combat anomalies, ore anomalies, and mission-combat dungeons.
- Choose solar system, stargate, station, celestial, or coordinate placement for server-owned content.
- Compose NPCs from server-side profiles/loadouts/behavior/loot/spawn groups.
- Override spawn wave count, trigger, delay, target rules, orbit/leash behavior, bounty, loot table, and fitted modules through server-authored rows.
- Define ore/gas/ice/resource composition by type IDs that already exist in the client/SDE.
- Define mission-combat templates and link them to agent/mission pools in a server-controlled overlay.

Disallowed without deeper client/SDE compatibility work:

- Editing client type definitions, ship attributes, dogma attributes, group/category metadata, icons, models, names expected by the client, or SDE-owned stargate/solar-system topology.
- Inventing new type IDs or ship hulls not present in the client.
- Regenerating `dungeonAuthority` or `missionAuthority` from SDE alone.
- Treating private mission sites as public scanner anomalies.
- Treating CONCORD Crimewatch punishment as passive gate scenery.

## Ore Anomaly Accuracy Requirements

Current EveJS clone state:

- `dungeonAuthority.templatesByID["client-dungeon:4200"]` can identify `Colossal Asteroid Cluster` as an ore anomaly, but `resourceComposition.oreTypeIDs` is only a typeID presence list. It does not carry rock counts, per-rock quantities, total units, layout, or depletion baselines.
- Existing public ore anomaly records appear to be metadata/scanner/template records. They are not enough by themselves to spawn mineable asteroid entities with accurate counts and volumes.
- Actual mineable entities use the mining runtime contract:
  - static entity fields such as `kind`, `itemID`, `typeID`, `slimTypeID`, `miningYieldTypeID`, `miningYieldKind`, `itemName`, `position`, `radius`, and `resourceQuantity`.
  - persisted mining state in `miningRuntimeState` with `entityID`, `yieldTypeID`, `yieldKind`, `unitVolume`, `originalQuantity`, `remainingQuantity`, and `originalRadius`.
- Existing regular asteroid belts already create `kind: "asteroid"` entities in `server/src/space/asteroids/asteroidService.js`. Mining runtime can consume those because `isMineableStaticEntity()` accepts `kind === "asteroid"` and `estimateOriginalQuantity()` honors explicit `resourceQuantity`.
- Existing generated mining sites in `server/src/services/mining/miningResourceSiteService.js` use a `members[]` shape with `entityID`, `childIndex`, `yieldTypeID`, `yieldKind`, `resourceName`, `visualTypeID`, `originalQuantity`, `remainingQuantity`, `originalRadius`, and `unitVolume`. That is the closest server-side model for authored anomaly resources.
- Generated mining universe seeding currently focuses on generated ice sites, plus dungeon gas materialization. Ore public dungeon templates need a new authored-ore materialization adapter if we want wiki-like asteroid anomaly layouts.

Recommended template shape for accurate ore anomalies:

- Keep `resourceComposition.oreTypeIDs` as a summary/index field for search/filtering/backward compatibility.
- Add an authored resource layout, e.g. `resourceLayout.members[]`, where each entry stores:
  - `typeID` / `yieldTypeID`
  - `typeName`
  - `asteroidCount`
  - `totalQuantity`
  - optional `perAsteroidQuantities[]`; otherwise distribute total quantity deterministically across count
  - `unitVolume` from `itemTypes`
  - optional `cluster`, `radiusMeters`, `spreadMeters`, `placementSeed`, and `variance`
  - `sourceURL`, `sourceName`, and `sourceConfidence`
- When applying the template to EveJS, expand `resourceLayout.members[]` into per-rock member/entity records. For the Colossal example, that means 58 asteroid entities total: 4 Arkonor, 7 Bistot, 9 Crokite, 6 Dark Ochre, 13 Gneiss, 5 Mercoxit, and 14 Spodumain.
- For close-enough layouts, generate stable per-rock quantities by splitting the wiki total over the count with deterministic +/-10-20% variance, then correcting the last rock so totals match exactly.
- For server integration, either:
  - add authored ore support to `miningResourceSiteService` using the same member/entity contract as generated mining sites, with `kind: "asteroid"` and `miningYieldKind: "ore"`;
  - or add ore-resource materialization to `dungeonUniverseSiteService.materializeSiteContents()` for `siteFamily === "ore"`.
- Prefer the first option because it keeps all mineable resource entity creation and mining runtime persistence in the mining service layer.
